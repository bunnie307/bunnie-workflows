# Strategy 파일 규격

이 문서는 strategy/ 디렉토리 내 모든 전략 파일의 공통 형식을 정의한다.

## 디렉토리 구조

```
strategy/
  [domain]/           # 도메인별 디렉토리 (testing, review, deploy, debug 등)
    [파일명].md       # 해당 도메인의 전략 파일
```

## 파일 형식

모든 strategy 파일은 다음 구조를 따른다:

```markdown
# [도메인] [관점/패턴] 목록

[이 파일의 목적 한 줄 설명]
[새 프로젝트 초기화 시 이 목록을 기본으로 사용한다는 안내]

## 기본 [관점/패턴]

### N. [이름] — [한 줄 설명]
- [구체적 확인 항목 1]
- [구체적 확인 항목 2]
- [구체적 확인 항목 3]

### N+1. [이름] — [한 줄 설명]
- ...

## 프로젝트 발견 [관점/패턴]

<!-- 실전에서 발견되어 자동 추가된 항목들. 아래 항목은 프로젝트 실전에서 진화 스킬이 발견하여 추가한 것이다. -->

### N. [이름] — [한 줄 설명]
- [구체적 확인 항목]
> 유래: YYYY-MM-DD [프로젝트명]에서 [구체적 사건 설명]
```

## 유래(Provenance) 형식

실전에서 발견된 모든 항목은 유래 정보를 반드시 포함한다:

```
> 유래: YYYY-MM-DD [프로젝트명]에서 [구체적 사건 설명]
```

예시:
```
> 유래: 2026-04-10 chain-indexer에서 API 응답 래퍼 불일치 버그
> 유래: 2026-04-10 chain-indexer에서 BigInt(undefined) 런타임 에러
```

## 도메인 간 교차 참조

다른 도메인과 겹치는 패턴이 발견되면:
- 각 도메인에 독립적으로 기록한다 (중복 허용)
- 유래에 관련 도메인을 언급한다

예시:
```
> 유래: 2026-04-10 chain-indexer에서 BigInt 직렬화 에러 (testing/Boundary 관점과 관련)
```

## 버전 관리

- 플러그인 레포의 strategy 파일이 정본이다
- 프로젝트별 추가 항목은 프로젝트 CLAUDE.md에 기록한다
- 프로젝트에서 새 관점이 발견되면 플러그인 레포에 동기화한다 (core/sync-rules.md 참조)
